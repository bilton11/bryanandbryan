1. Create `app/assessment/routes.py` implementing the HTMX wizard pattern from research:

   ```python
   from __future__ import annotations
   ```

   Define WIZARD_STEPS list: `["dispute_type", "facts", "amount", "opposing_party", "summary"]`
   
   Define STEP_LABELS dict mapping step names to human-readable labels for the progress bar.

   **Route: GET /assess (wizard entry point)**
   - `@login_required`
   - Get or create a DRAFT claim for current_user (query for existing draft, or create new)
   - Render `assessment/wizard_shell.html` with the claim's current_step partial embedded via `include`

   **Route: GET+POST /assess/step/<step_name> (wizard step handler)**
   - `@login_required`
   - Validate step_name is in WIZARD_STEPS
   - Get the user's draft claim (abort 404 if none)
   - On POST:
     - Call step-specific validation function
     - If errors: return the step partial with errors dict
     - If valid: persist step data to `claim.step_data[step_name] = dict(request.form)`, advance `claim.current_step` to next step, `db.session.commit()`
     - Return the next step partial (or redirect to summary if last step)
   - On GET:
     - Return the step partial template (for HTMX) OR the full wizard_shell (for direct URL access)
     - Detect HTMX with `request.headers.get("HX-Request") == "true"` -- if HTMX, return partial only; if not HTMX, return wizard_shell with the step embedded

   **Validation functions (private helpers):**
   - `_validate_dispute_type(form_data)`: Check claim_type is in [v["value"] for v in VALID_CLAIM_TYPES]. Then check against BOTH excluded and redirected types -- both are HARD STOPS that prevent the user from proceeding:
     - If claim_type is in EXCLUDED_CLAIM_TYPES: return error dict with message "This type of claim cannot be filed in Ontario Small Claims Court." and a generic explanation of the correct forum.
     - If claim_type is in REDIRECTED_CLAIM_TYPES: return error dict using the specific redirect message from the REDIRECTED_CLAIM_TYPES dict (which contains the plain-language forum explanation, e.g. LTB for landlord-tenant, Superior Court for defamation). This is a HARD STOP -- the user cannot proceed past this step. Do NOT allow proceeding with an acknowledgment checkbox.
     - Both EXCLUDED and REDIRECTED types return an error dict (not a warning). The step partial re-renders with the error displayed. The user must select a different claim type to continue.
   - `_validate_facts(form_data)`: Require `description` (min 20 chars), `incident_date` (valid date, not future)
   - `_validate_amount(form_data)`: Require `amount` (integer > 0, <= SMALL_CLAIMS_MONETARY_LIMIT from ontario_constants). Note: $50K jurisdiction check is a hard stop in this step.
   - `_validate_opposing_party(form_data)`: Require `party_name` (min 2 chars), `party_type` (individual or business)
   - `_validate_summary()`: No validation needed -- summary step is read-only review

   **Helper: _get_or_create_draft(user_id)**
   - Query `Claim.query.filter_by(user_id=user_id, status=ClaimStatus.DRAFT).first()`
   - If none exists, create a new Claim with user_id, commit, return it

   **Route: POST /assess/back/<step_name> (back navigation)**
   - Move claim.current_step to previous step
   - Return previous step partial
